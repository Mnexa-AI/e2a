# Async Send — Pre-GA API Contract Spec

**Status:** Approved direction, contract must land before the /v1 GA freeze (~mid-July 2026).
**Scope:** Contract/spec changes for the async send pipeline. **Decided 2026-07-03: the pipeline itself is a pre-GA blocker.** e2a's product contract is at-least-once outbound delivery, and the current send-then-persist path cannot honor it — there is no durable acceptance point before the network I/O, and the idempotency key only `Complete`s after the response (`internal/httpapi/idempotency.go:115`), so every timeout/5xx is ambiguous and unsafely retryable. GA ships on the target shape — API → durable persist+enqueue in one tx → 200 `accepted` → worker pool → SMTP (`async-message-pipeline.md` slices 1–3); slices 4–6 stay post-GA. A synchronous server still trivially satisfies this contract during the transition (it just never returns `accepted`), but the /v1 freeze happens only once the async path is the default, so callers GA onto the semantics they will actually live under.

**Background (one paragraph):** Today `POST /v1/agents/{email}/messages` submits to SES synchronously inside the request and returns 200 `sent` / 202 `pending_review`. Decision: move to durable acceptance — the API validates, renders, policy-checks, and persists in one transaction, then returns immediately with `status: accepted`; a worker pool later performs the content scan, ramp reservation, and SES submission. Every terminal outcome is observable via GET and pushed via webhook events. See the delivery-semantics audit for the crash-window motivation (send-then-persist loses/bills invisible messages).

## 1. Send response contract (`sendMessage`, `replyToMessage`, `forwardMessage`)

1.1. Add `accepted` to the send-result `status` known-values: `accepted | sent | pending_review | review_approved | failed` (the existing values stay; `accepted` becomes the primary success once the pipeline ships). `failed` is included because a `wait=sent` call can observe a terminal failure inside the window (§2.1) and callers must be able to parse it. Keep this an **open-set** (doc-string known-values, not a closed generated enum) so post-GA additions don't break pinned SDKs — same discipline as the delivery-status vocabulary (§3.1) and the request-side events list (§4.4a).
1.2. `accepted` responses: HTTP **200**, body includes `message_id` (minted pre-send, durable) and `status: "accepted"`. `provider_message_id` is documented as **optional/absent** until the message is sent — mark the field nullable/omittable in the Huma types so the OpenAPI spec and generated SDKs are correct. (**200 not 202** is a deliberate choice — 200 iff the request is durably persisted, §1.5; the single-200 shape keeps `body.status` the one discriminator across accepted/sent/failed/pending_review/timeout, rather than making callers branch on 200-vs-202. 202 is retired as a GA success code — legacy alias only. Document the rationale so it reads as intentional, not an HTTP-semantics slip.)
1.3. Document semantics precisely: `accepted` = "durably persisted and queued for submission; e2a attempts delivery until the provider accepts or a terminal failure is declared — the outcome arrives via webhook events (`email.sent` / `email.failed`) or `GET /v1/messages/{id}`. An accepted message is never silently dropped." `sent` = "accepted by the upstream provider" (NOT delivered).
   **Every accepted send reaches exactly one terminal push signal — freeze this now.** The §1.3 promise ("outcome arrives via `email.sent`/`email.failed`") must hold for *every* accepted send, including ones that are never sent. A send that is held for review and then **review-rejected, expired, or scan-blocked** (post-slice-4, when block/hold become async) is never sent: without care it gets no `email.sent`, no `email.failed`, and its `delivery_status` rests at `accepted` forever — a caller that implements §1.3 literally ("wait for sent/failed, or poll to a terminal `delivery_status`") hangs forever. **Decision (frozen at GA):** block/reject/expiry emits a terminal `email.failed` carrying a `reason` (`blocked` / `review_rejected` / `review_expired`) — one terminal push signal for all "accepted but will never send" outcomes — and `GET` reflects a terminal `delivery_status` (`failed`), never a permanent `accepted`. This text is frozen even though the async-block implementation is slice 4 (post-GA), because §1.3's contract cannot be amended after the freeze.
1.4. Synchronous failures remain synchronous (these all run before the persist): 400 validation / `template_render_failed`, 403 `blocked_by_policy` (recipient-policy gate) / `quota_exceeded` / unverified domain, 422 `recipient_suppressed`, 429 rate limit, 413 body too large.
1.5. **Contract change to flag loudly in docs/changelog (for current beta users):** the default success response flips from 200 `sent` to 200 `accepted` when async becomes the default — this now happens BEFORE the GA freeze (pipeline slice 3), so there is no post-freeze flip of the primary status.
   **Decided (owner, 2026-07-04): 200 iff the API request is durably persisted — persistence is the discriminator, not the eventual outcome.** So a review-**held** send, once its (durable) hold row is committed, returns **200** with `body.status: "pending_review"` — NOT 202. This retires 202 as a success code at GA (it remains only as a documented legacy alias for the beta→GA transition), and makes `body.status` the single value callers branch on across `accepted | sent | pending_review | failed` (§2.1 already states "the HTTP status code is not the outcome discriminator"). Because holds are 200 from GA onward, slice 4's move of scanning into the worker changes only *when* the hold is known (request-time `pending_review` while the scan is sync at GA → request-time `accepted` + a following `email.pending_review` event once async), never the HTTP code — no breaking status-code flip after the freeze. Callers are told from GA: treat any 2xx as "persisted; read `body.status`."

## 2. `wait` parameter (sync-compat valve)

2.1. Add optional query param `wait=sent` to the send endpoints: server holds the request until the message reaches a terminal-or-held state or a bounded timeout, then returns that state. **Every non-timeout outcome must have a frozen (HTTP code, `body.status`) pair — an underspecified `wait` is the most expensive thing to leave ambiguous at freeze.** The per-outcome table (frozen):

   | Outcome within the wait window | HTTP | `body.status` | notes |
   |---|---|---|---|
   | Sent | 200 | `sent` | `provider_message_id` present |
   | Terminally failed | 200 | `failed` | carries `delivery_detail`; **`failed` must be in the §1.1 send-status known-values list** or `wait` callers can't parse their own success case |
   | Held for review | 200 | `pending_review` | 200 = persisted (§1.5); the durable hold row exists. NOT 202 |
   | Timeout (still in flight) | 200 | `accepted` | the fast-path shape; caller polls `GET` or waits for the event |

   **Rule, stated in the contract: the HTTP status code is NOT the outcome discriminator — always branch on `body.status`** (200 covers accepted/sent/failed/held/timeout — every persisted outcome). 
2.2. Default is NO wait (fast path). A synchronous server implements `wait` as a no-op (it always has the outcome).
2.3. **Bound the wait ceiling explicitly (frozen): at most 20s**, so clients and proxies (ALB/ingress idle timeouts) can size their own timeouts against a known ceiling. A long-poll that outlives a proxy timeout surfaces as a client error for a send that is actually succeeding.
2.4. **`wait` + idempotent replay:** a replay of a *completed* key returns the cached accept-time response (necessarily `accepted`, since `Complete` commits in the accept-tx) regardless of `wait` — it does NOT re-wait; the caller polls `GET` for current state. A same-key retry arriving *during* another request's wait gets `409 idempotency_in_flight` (specify this as the expected answer, and note SDKs must treat it as retry-after, not a hard error — the TS SDK currently retries only 408/429/5xx).

## 3. Message lifecycle model

3.1. Delivery-status vocabulary on `GET /v1/messages/{id}` (`delivery_status`): **decided — use `accepted`** (same word as the send-response status; one vocabulary for one state, not two names forever). The dead `queued` value (`internal/delivery/status.go:14`, never emitted in production) is dropped from the docs. Add `sending` (worker lease held). Full set: `accepted → sending → sent → delivered | deferred | bounced | complained | failed`. Update the open-set doc string in `internal/httpapi/messages.go` and extend the precedence chain in `internal/delivery/status.go` (`accepted`/`sending` below `sent`). While a message is held for review, `delivery_status` stays `accepted` — the hold is visible on `messages.status` (`pending_review`); the two lifecycles keep their own columns. **Reserve `blocked` in the `messages.status` open-set doc string now** (documented but not emitted until slice 4) so the post-GA async-block outcome (§1.3) isn't a vocabulary addition after the freeze. The `delivery.Merge` precedence gets one **exception: header-matched provider `sent`/`delivered` feedback overrides a local `failed`** (else a falsely-declared terminal `failed` from a final-attempt crash is uncorrectable — see pipeline §7).
3.2. Failed semantics: `failed` = worker exhausted retries or SES rejected permanently; carries `delivery_detail`.

## 4. Webhook events

4.1. Add **`email.failed`** event (terminal async failure). Required — with instant-200, events are the only push signal that a send died. Same envelope as `email.sent`; include `message_id`, `recipient`(s), `detail`, `retryable: false`, and `reason` (covers block/reject/expiry per §1.3).
4.1a. **Subscription gap — `email.failed` is opt-in but load-bearing.** Webhook subscriptions require an explicit non-empty event list with no wildcard (`webhooks.go:93-99`), so every subscriber created during beta — everyone subscribed to `email.sent` — is NOT subscribed to `email.failed` and, the moment async becomes the default, receives no push signal that a send died. **Decide before freeze:** either (a) support a `*`/`email.*` wildcard subscription, (b) auto-add `email.failed` to every existing and new subscription that includes `email.sent` (paired events), or (c) make terminal failure non-filterable (always delivered). Recommendation: (b) at minimum, backfilled by migration. Without one of these, "`email.failed` required" is true in the spec and false in every deployed subscription.
4.2. Add **`email.deferred`** event. **Decided: add it** (was "decide" — a freezing contract cannot leave an event type undecided; both Resend and SendGrid push a deferred/delayed signal, and agents poll poorly). Mark it beta if unsure of the shape, but reserve the type at GA.
4.3. Mark stable-or-beta consistently with the webhook-events GA curation (disposition events precedent). `email.failed` is stable (it's "required"); `email.deferred` may ship beta.
4.4. All send-related events gain an optional `batch_id` correlation field (empty for single sends) — pre-wiring for batch send.
4.4a. **Request-side event enum must be open, or SDKs can't subscribe to post-GA events.** Response-side event *types* are correctly open-set (doc-string strings), but the **request** side — `Create/UpdateWebhookRequest.events` and `TestWebhookRequest.event` — carries `enum:` tags that OpenAPI Generator turns into a *closed* TS union. A pinned GA TypeScript SDK then cannot subscribe to any event type added after the freeze (e.g. a later `email.deferred` or `email.blocked`) without an SDK bump. Before freeze: make the request-side `events` a validated open string (accept unknown types, or validate server-side against the live registry rather than a frozen enum in the generated types).
4.5. **Durable event emission (ties at-least-once end-to-end):** `email.sent`/`email.failed` are the only push signals of an async outcome, so they must not ride the fire-and-forget legacy publish goroutine (`go publisher.Publish(...)`, which swallows errors). The worker emits them via the webhook outbox (`PublishTx`) in the same transaction that marks the message sent/failed. **The durable outbox is mandatory whenever `outbound.mode=async`, not a separate flag** (pipeline §10): async mode implies the outbox is the event path — startup refuses `async` + legacy-publisher config — so a self-hoster (the open-core deployment the single-binary constraint exists for) cannot silently run async with `WEBHOOKS_OUTBOX_ENABLED` off and lose every failure event. Self-host default is `outbound.mode=sync`; opting into async turns the outbox on with it. Hosted (e2a-ops) ships async+outbox at GA. The outbox machinery (same-tx `webhook_events` write, deterministic event ids, drain worker) is already built and tested.
4.6. **Scope of the at-least-once guarantee at GA: direct sends.** The approve/HITL path publishes outcomes via the legacy fire-and-forget publisher (`hitlworker`) until pipeline slice 5 unifies it onto the queue. Either pull approved-hold outcome events onto the outbox in slice 3, or state explicitly that §4.5's durability covers direct sends at GA and approved-hold events are best-effort until slice 5 — do not let the docs imply a durability the approve path doesn't yet have.

## 5. Idempotency contract

5.1. Document on the send endpoints: `Idempotency-Key` strongly recommended; same key + same body → same `message_id`, never a second send. The pipeline (slice 2) makes this hold by committing `Complete` **inside the durable acceptance transaction** — the accept-tx for allowed sends AND `HoldForApprovalCore`'s insert for review-held sends (a held send never reaches the accept-tx; pipeline §6). Today `Complete` runs after the response (`internal/httpapi/idempotency.go:115`), so a crash after SES-accept re-executes on retry and double-sends — the "5-min stale-window caveat" undersold the gap.
5.1a. **The frozen contract must state the semantics that already exist in code (don't freeze a vaguer contract than the implementation):**
   - **Scope:** keys are one flat per-account namespace across ALL idempotent endpoints (`(user_id, 'u:'+key)`; the stored hash covers route + raw body, `idempotency.go:37-40`). So reusing one key on `send` then `reply` is a body/route *mismatch*, not two sends — document that a key is per-account and the match includes the endpoint+body.
   - **Body mismatch:** same key + *different* body → **422 `idempotency_key_reuse`**, not a silent second send. 422 (Unprocessable Content), not 409: the request is well-formed but the `(key, body)` pairing violates the idempotency contract — this matches the IETF Idempotency-Key header draft, which specifies 422 for a payload/fingerprint mismatch. (409 is reserved for the genuine *concurrency* conflict below.)
   - **In-flight concurrency:** same key while the first request is still running → **409 `idempotency_in_flight`** (a real conflict — a second request racing the first; also the `wait`-window answer, §2.4).
   - **TTL — the honest bound:** completed keys are swept after **~24h** (`idempotency` sweep). So "never a second send" holds *within the key's retention window*; a retry with the same key after 24h is a new operation and CAN send again. State the window; "unqualified forever" is false. This is standard (Stripe documents 24h; Resend documents its own window) — match peer disclosure, don't overclaim.
5.2. SDK follow-up (not spec, but note): TS/Python SDKs auto-mint a UUID key per send call unless caller supplies one — and must NOT internally retry a `409 idempotency_in_flight` as a failure (it means the original is still succeeding); treat 409-in-flight as retry-after.

## 6. CLI exit-code contract (frozen — additive only)

6.1. `e2a send`/`reply` default to `wait=sent` so exit codes keep meaning: 0 = sent, 3 = held (non-sent status). Do NOT let a bare `accepted` map to exit 3.
6.2. Reality check: the CLI has no `send` command yet (today: login/listen/config; `cli/src/exit.ts` does not exist — it arrives with the CLI harness-primitives plan). Nothing is frozen, so define the contract accepted-native from day one rather than retrofitting. The full map, needed at once (every `wait` outcome in §2.1 must have an exit code, or the timeout/failed cases map to an accidental nonzero and break scripts): **0 = sent, 3 = held (`pending_review`), 4 = accepted/timeout (in flight, not an error — must NOT be nonzero-as-failure), 5 = failed (terminal).** `--no-wait` returns 4. Never renumber once shipped.

## 7. Two pre-GA correctness fixes (server behavior, not spec — ship together)

7.1. `internal/agent/api.go` (~1271): `CreateOutboundMessage` failure is currently logged and swallowed while still returning 200 — make it a 500 (send happened but is unrecorded; do NOT claim clean success). Post-pipeline this becomes impossible by construction.
7.2. Move `recordOutboundUsage` (~1269) to AFTER the message row persist so a crash can't bill a message that has no row.
7.3. Interaction to note: 7.1 alone converts today's silent loss (sent + unrecorded + 200) into duplicate-on-retry (sent + 500 → caller retries → second send, because the idempotency key never `Complete`d). Ship it anyway — honest failure beats silent loss — but it is a stopgap; persist-first (pipeline slice 2) removes the window by construction.

## 8. Process

8.1. Every /v1 handler change here → `make spec` + commit `api/openapi.yaml`, `make generate-sdk` + commit generated SDK bases (CI drift gates: spec-check, generate-sdk-check).
8.2. Contract tests: add cases asserting the `accepted` status is a valid response shape, `provider_message_id` optionality, and `wait=sent` behavior (no-op on sync server).
8.3. Docs: update `docs/api.md` + `docs/events.md` with the lifecycle state machine and the "sent ≠ delivered" clarification.
